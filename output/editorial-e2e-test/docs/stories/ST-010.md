# ST-010 — Implement draft and publish workflow

**Story key:** `feature.draft-publish`

## Description

Add draft, review, and publish states with role-based transitions.

## Acceptance Criteria

- [ ] Collections intended for public rendering (posts, pages) support editorial states `draft`, `in_review`, and `published`
- [ ] Editors can save drafts and reviewers/admins can transition content to published state (admin or reviewer)
- [ ] A draft can be sent back from review for changes without creating duplicate entries
- [ ] The query/filter layer used by future public rendering excludes draft entries unless preview/draft mode is explicitly active
- [ ] Publishing records a canonical published timestamp or Payload published-state equivalent used by later public rendering
- [ ] Database migration is generated and executed for all schema changes (run `npm run db:migrate:create` then `npm run db:migrate`)
- [ ] Publishing requires admin or reviewer role

## Scope Boundaries

- Do NOT implement versioning or revision history
- Do NOT implement scheduled publishing — that is a separate story
- Do NOT implement approval notifications
- Do NOT implement public page components in this story — only the workflow and publish visibility rules
- All migrations MUST be created in `src/lib/migrations/` using `npm run db:migrate:create` — do NOT create migration files in any other directory (e.g. `src/models/migrations/`)

## Expected Files

- `src/lib/content-status.ts`
- `src/collections/Posts.ts`
- `src/collections/Pages.ts`

## Dependencies

- `product.cms-content-model`
- `feature.roles`

## Validation

- Run: `npm run build`
- Run: `npm test`
- Manual: Can create a draft, submit/review it, publish it, and confirm only published entries are returned by the public-facing query/filter path.
