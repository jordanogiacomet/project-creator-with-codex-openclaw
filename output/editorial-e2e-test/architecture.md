# Architecture

**Style:** service-oriented

## Components

### cdn
- **Technology:** cdn
- **Role:** public asset delivery

### frontend
- **Technology:** nextjs
- **Role:** user interface

### api
- **Technology:** payload
- **Role:** application logic

### database
- **Technology:** postgres
- **Role:** persistent storage

### object-storage
- **Technology:** s3-compatible
- **Role:** media storage

### worker
- **Technology:** payload
- **Role:** background processing

## Communication

### frontend → api
- **Protocol:** http
- **Pattern:** REST via Payload auto-generated API + custom endpoints
- **Auth:** JWT bearer token or session cookie (Payload built-in)

### api → database
- **Protocol:** tcp
- **Pattern:** Drizzle ORM (Payload built-in) over postgres connection pool

### api → object-storage
- **Protocol:** http
- **Pattern:** S3-compatible API for media upload and retrieval
- **Auth:** Access key / IAM role

### worker → database
- **Protocol:** tcp
- **Pattern:** Direct postgres connection for job execution

### cdn → frontend
- **Protocol:** http
- **Pattern:** Edge cache proxying static assets and SSR/ISR responses

## Responsibility Boundaries

### Frontend
- Rendering pages and UI components
- Client-side routing and navigation
- Form validation (client-side)
- Client-side state management
- Public page rendering (SSR/ISR) for SEO
- CDN-compatible caching headers

### Backend
- Business logic and domain rules
- Authentication and session management
- Authorization and permission enforcement
- Data persistence and queries
- Input validation (server-side, authoritative)
- Content model definition via Payload collections
- Media upload handling and storage
- Media asset management and storage orchestration

### Shared
- TypeScript types auto-generated by Payload

## Typical Request Flow

### Authenticated action (e.g., create or update a record)

1. User interacts with the frontend (nextjs)
2. Frontend sends HTTP request to the API (payload) with auth token/session
3. API middleware validates authentication and authorization
4. API executes business logic and validates input
5. API persists data to postgres via ORM/query layer
6. API returns response to frontend
7. Frontend updates UI based on response

### Background job execution

1. Scheduler triggers job at configured interval
2. Worker queries postgres for pending work
3. Worker processes each item (publish, notify, remind, etc.)
4. Worker updates postgres with results
5. Worker logs outcome for observability

## Key Constraints

- Database: `postgres` — all migrations in `src/lib/migrations/`
- Backend: Payload CMS — collections in `src/collections/`, admin at `/admin`
- Environment: use variable names exactly as in `.env.example`
- Router: App Router only (`src/app/`) — do NOT use `src/pages/`
- See `decisions.md` for full architectural decisions
