# V0 Boundary

## Purpose

This document freezes the operational boundary for V0 of the **OpenClaw Project Initializer**.

V0 exists to turn a natural-language project idea into a reviewable bootstrap package for one supported archetype. It must reduce ambiguity, force missing architecture questions into the open, generate the initial project starting structure, and stop for human review before any implementation begins.

## Supported Archetype

V0 supports exactly one archetype:

- `Next.js + Payload + PostgreSQL`

V0 should support this archetype well rather than attempt broad multi-stack coverage.

## Expected Input Model

V0 accepts a local CLI invocation through the planned command:

```bash
initializer new
```

The user provides:

- a free-text natural-language project description;
- confirmation or correction of the detected archetype;
- answers to guided follow-up questions for missing architectural context;
- explicit confirmation of critical architecture decisions before generation.

Strong defaults in V0 are:

- database: PostgreSQL
- deploy target: Docker
- validation: lint, typecheck, build

## Expected Output Model

V0 generates a reviewable bootstrap package in a new directory.

Generation in V0 is limited to docs, structure, and initial stories only. Supporting bootstrap files such as `README.md`, `AGENTS.md`, `progress.txt`, `.env.example`, and `docker-compose.yml` are part of that generated bootstrap structure.

The generated output is expected to provide:

- project definition artifacts such as `PRD.md` and `prd.json`;
- repository operating documents such as `decisions.md`, `progress.txt`, `README.md`, and `AGENTS.md`;
- bootstrap structure and operational scaffolding;
- an initial story backlog for later Codex + Ralph loop execution.

## V0 Limits

V0 must:

- remain local-first;
- ask for missing information needed to create a coherent bootstrap;
- require explicit confirmation for critical architecture decisions;
- stop after generation and before implementation.

V0 must not:

- implement the generated project's business features;
- continue automatically into Codex + Ralph loop execution;
- write into an existing project directory by default;
- expand beyond the supported archetype without a later approved decision.

## Handoff Boundary

The V0 boundary ends when the initializer has generated the bootstrap package and presented it for human review.

Only after explicit human approval may the generated repository move into story-based implementation with Codex + Ralph loop. Discovery and bootstrap generation belong to the initializer; feature implementation does not.

## Explicit Non-Goals

V0 explicitly excludes:

- deploy automation;
- production access or production changes of any kind;
- automatic feature implementation;
- parallel story execution;
- silent architecture assumptions;
- broad multi-archetype support.

## Minimal Execution Flow

1. The user runs `initializer new`.
2. The user provides a natural-language project idea.
3. The initializer detects the `Next.js + Payload + PostgreSQL` archetype.
4. The initializer asks guided follow-up questions and confirms critical architecture decisions.
5. The initializer consolidates the resolved project context.
6. The initializer generates the bootstrap docs, structure, and initial stories into a new directory.
7. The initializer stops for human review before any implementation work begins.
